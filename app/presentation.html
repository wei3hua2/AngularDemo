Stateful vs Stateless Server

  • The server keeps the necessary state in memory of the client state of the page.

  • When any request hits the server (usually user actions), the server sends the appropriate HTML and/or JavaScript with the concrete changes to bring the client to the new desired state. At the same time the state in server is updated.

  • Most of the logic is executed in server and HTML is usually also rendered in server. In some ways the server simulates a web browser, receiving events and performing delta changes in server state which are automatically propagated to client.

Stateful vs Stateless Server

  • This approach needs more server memory and server processing.

  • Has a mix of concern with the presentation and business logic.

  • The advantage is a simplified development model because:

  • The application is usually fully coded in server.

  • Data and UI state in server are shared in the same memory space with no need of custom client/server communication bridges.

Stateful vs Stateless Server

  • The client page sends the necessary data of the current state to the server usually through AJAX requests.

  • The server with this data is able to reconstruct in some way the client state of the page part going to be modified and generate the necessary data or code.

Stateful vs Stateless Server

  • This approach is more easily scalable because there is no per client page data kept in server and therefore AJAX requests can be dispatched to different server nodes with no need of session data sharing or server affinity.

  • Well adopted by REST architectural style. Clear separation of concern between business logic (server) and presentation (SPA).

Single Page Application


  • A web application with the goal of providing a more fluid user experience akin to a desktop application.

  • All necessary code – HTML, JavaScript, and CSS – is either retrieved with a single page load, or the appropriate resources are dynamically loaded and added to the page as necessary.

  • Typically rely on a stateless RESTFUL approach to communicate with the server.

Single Page Application


  • Clear separation of concern encourages reuse for business logic (functional requirement). It also creates an abstraction layer for testing.

  • Browser has become the new Virtual Machine. For mobile devices, browser engine is used as the mediator, cordova is used as an abstract layer to communicate with native feature if need to.

  • A shared common language (Javascript) is used across different platforms and devices.

Angular JS Architecture

Philosophy & Principle
  • Declarative code is better than imperative when it comes to building UIs and wiring software components together, while imperative code is excellent for expressing business logic.

  • Decouple DOM manipulation from application logic. This improves the testability of the code.

  • Regard application testing as equal in importance to application writing.

Angular JS Architecture

Philosophy & Principle
  • Follows the MVC pattern of software engineering and encourages loose coupling between presentation, data, and logic components.

  • Using dependency injection, testability improves & mock data is easily injected.

Angular JS Architecture

remove boilerplates by freeing us from..
  • Registering callbacks: Registering callbacks clutters your code, making it hard to see the forest for the trees. Removing common boilerplate code such as callbacks is a good thing. It vastly reduces the amount of JavaScript coding you have to do, and it makes it easier to see what your application does.

  • Manipulating HTML DOM programmatically: Manipulating HTML DOM is a cornerstone of AJAX applications, but it's cumbersome and error-prone. By declaratively describing how the UI should change as your application state changes, you are freed from low-level DOM manipulation tasks. Most applications written with Angular never have to programmatically manipulate the DOM, although you can if you want to.

Angular JS Architecture

remove boilerplates by freeing us from..
  • Marshaling data to and from the UI: CRUD operations make up the majority of AJAX applications' tasks. The flow of marshaling data from the server to an internal object to an HTML form, allowing users to modify the form, validating the form, displaying validation errors, returning to an internal model, and then back to the server, creates a lot of boilerplate code. Angular eliminates almost all of this boilerplate, leaving code that describes the overall flow of the application rather than all of the implementation details.

  • Writing tons of initialization code just to get started: Typically you need to write a lot of plumbing just to get a basic "Hello World" AJAX app working. With Angular you can bootstrap your app easily using services, which are auto-injected into your application in a Guice-like dependency-injection style. This allows you to get started developing features quickly. As a bonus, you get full control over the initialization process in automated tests.

Model-View-Controller


Angular has a flexible model definition.

Each view is defined by a scope, where model attributes are passed.

Every view has a tight one-to-one relationship with a controller.

Model is automatically updated when there is an update on either controller or view, known as two-way binding.

This free up developer from having to manually updating the view with the model attributes.

Dependency Injection


  • The advantage of DI comes from the declaration of dependency only when it is needed.

  • An injector (container) will then take care of instantiation of the dependencies.

  • Thus, the responsibility of locating the dependency is shifted from the depender to the injector container. Dynamically resolve the dependencies during runtime.

  • Making testing of the component easier.
Factory pattern to generate singleton object
            
myModule.factory('greeter', function($window) {
    return {
        greet: function(text) {
            $window.alert(text);
        }
    };
});
            
Declaring greeter on another module
            
myModule.controller('GreetingCtrl',function($scope, greeter) {
    $scope.greeting = function () {
        greeter.greet('Hello World');
    }
});

Module: component-based architecture

  • Angular JS is a component-based architecture.

  • There is no main method that instantiates and wires together the different parts of the application.

Module: component-based architecture


The advantage of this approach includes:


  • You can package code as reusable modules.

  • The modules can be loaded in any order (or even in parallel) because modules delay execution.

  • Unit tests only have to load relevant modules, which keeps them fast.
The module 'xmpl' includes 3 other modules.
            
angular.module('xmpl.service', []).
    value('greeter', {
        salutation: 'Hello',
        localize: function(localization) {
            this.salutation = localization.salutation;
        },
    greet: function(name) { return this.salutation + ' ' + name + '!'; }
    }).value('user', {
            load: function(name) {
            this.name = name;
       }
    });
